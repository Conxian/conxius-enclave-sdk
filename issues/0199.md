# [P1] Make toolchain, dependencies, and release evidence reproducible and single-path

**Issue #199** | OPEN | Created: 2026-07-20 | Updated: 2026-07-20

The audit records release, toolchain, dependency, and provenance inconsistencies, including crate metadata at 2.0.12 versus visible v2.0.11 release evidence. Without a reproducible dependency graph and durable artifact evidence, the SDK cannot make or support a stable production release claim.

Refs #191

## Scope
- Define and enforce one supported Rust/MSRV policy and one reproducible dependency/lockfile policy for the SDK.
- Reconcile `Cargo.toml` metadata, tags, registry publication, changelog, and release evidence.
- Make `crates-publish.yml`, `release.yml`, `release-strict.yml`, and `secret-scan.yml` single-path, deterministic, and mutually consistent.
- Ensure SBOM, provenance, secret-scan, and exact-artifact evidence is retained and reviewable without exposing sensitive operational data.

## Evidence
- [Audit PR #193](https://github.com/Conxian/conxius-enclave-sdk/pull/193) identifies dependency/toolchain, release provenance, and CI scanning gaps.
- Historical issues #145, #154, #169, and #176 cover earlier CI, release, provenance, and version failures.
- Merged [PR #177](https://github.com/Conxian/conxius-enclave-sdk/pull/177) removed a duplicate publisher path, while merged [PR #178](https://github.com/Conxian/conxius-enclave-sdk/pull/178) moved metadata to 2.0.12; these are related evidence, not proof that the complete release path is reproducible.

## Acceptance criteria
- Exactly one automatic publisher and one release creator exist for each supported release path, with documented manual recovery boundaries that do not create competing publishers.
- The supported toolchain and dependency graph are pinned, reproducible, and verified against the declared MSRV; lockfile policy is explicit.
- Version, tag, registry artifact, changelog, checksums, SBOM, and provenance identify the same exact release candidate.
- Secret scanning is a real enforced check over the intended history/artifacts, not a placeholder or documentation-only step.
- Exact-artifact provenance and verification results are retained long enough to support independent release review and rollback decisions.

## Dependencies
- Can proceed alongside implementation work, but must be complete before the final acceptance gate in this backlog.
- Release evidence must cover only capabilities that pass the hardware, canonical verification, protocol, and asset gates in #195, #196, #197, and #198.
