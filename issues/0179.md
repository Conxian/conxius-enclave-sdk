# [BIP-110] Add bip110_compliant feature flag

**Issue #179** | OPEN | Created: 2026-07-15 | Updated: 2026-07-15

## Summary

Add a `bip110_compliant` feature flag to ensure the SDK works correctly under BIP-110 (Reduced Data Temporary Softfork) rules.

## Context

BIP-110 limits data embedding in Bitcoin transactions:
- Max 256-byte pushdata
- 83-byte OP_RETURN
- 34-byte ScriptPubKey

See [docs/BIP110_ALIGNMENT.md](https://github.com/Conxian/lib-conxian-core/blob/main/docs/BIP110_ALIGNMENT.md) for full guidance.

## Requirements

- [x] Add `bip110_compliant` feature flag to Cargo.toml
- [ ] Verify BIP-322 works under new rules
- [x] Add compliance tests for transaction builders (basic tests in src/protocol/bip110.rs)
- [x] Document max data sizes in signing operations

## Priority

**HIGH** - This directly impacts Bitcoin transaction handling.

## Implementation

- Feature flag: `bip110_compliant` added to Cargo.toml
- Module: `src/protocol/bip110.rs` with Bip110Validator
- Limits: 256-byte pushdata, 83-byte OP_RETURN, 34-byte ScriptPubKey
- Tests: 9 unit tests for validation and chunking

## Remaining Work

1. Integrate BIP-110 validation into BIP-322 message signing
2. Add compliance tests for Ark/BitVM2 data commitment segmentation
3. Document signing operation limits
