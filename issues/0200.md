# [P1] Harden the WASM secret boundary and add runtime/platform evidence

**Issue #200** | OPEN | Created: 2026-07-20 | Updated: 2026-07-20

The audit identifies build-only WASM lanes, hardware mock paths, and missing runtime/platform evidence as P1 gaps. Without an opaque secret boundary and evidence for supported runtimes, browser-facing APIs cannot be treated as production-supported even when the Rust build succeeds.

Refs #191

## Scope
- Harden `src/wasm_bindings.rs` so private signing material remains behind opaque handles or approved operations and is never returned through JavaScript-facing APIs.
- Add runtime evidence for browser, Node, bundler, and worker environments rather than relying only on build-only CI lanes.
- Separate hardware mocks from production attestation evidence and prevent mocks from satisfying production trust requirements.
- Make unsupported adapters and platforms return typed fail-closed outcomes.

## Evidence
- [Audit PR #193](https://github.com/Conxian/conxius-enclave-sdk/pull/193) records WASM/platform/hardware evidence gaps as P1 findings.
- Historical issue #172 audited missing WASM module bindings, but completeness alone does not establish a secure secret boundary or runtime/platform support.
- Relevant implementation and CI surfaces include `src/wasm_bindings.rs`, WASM build lanes, and hardware mock paths.

## Acceptance criteria
- JavaScript-facing APIs expose only opaque references, public results, or approved operation results; private keys and equivalent secret material cannot be read back.
- Browser, Node, bundler, and worker tests cover initialization, signing requests, error paths, lifecycle, and unsupported-platform behavior.
- A public support matrix identifies tested runtime/platform combinations and explicitly marks conditional or unsupported combinations.
- Hardware mocks cannot satisfy production hardware or attestation policy checks.
- Unsupported adapters fail closed and malformed boundary inputs return typed errors without panic or silent fallback.

## Dependencies
- Depends on the hardware-backed boundary in #195 and coordinates with canonical signing/verification in #196.
- Must be attached to the final acceptance evidence before any WASM capability is described as stable.
