# [P0] Enforce hardware-backed signing and mandatory attestation for value-bearing operations

**Issue #195** | OPEN | Created: 2026-07-20 | Updated: 2026-07-21

The audit recorded by [PR #193](https://github.com/Conxian/conxius-enclave-sdk/pull/193) identifies simulated/software signing paths and incomplete attestation enforcement as P0 production blockers. Until value-bearing operations require hardware-backed keys and complete evidence checks, the affected capabilities must remain conditional and cannot be enabled for production.

Refs #191

## Scope
- Enforce the production behavior of the default `unlock` path in `src/enclave/mod.rs` so a value-bearing operation cannot instantiate a simulated signer.
- Harden `src/enclave/android_strongbox.rs` and `CoreEnclaveManager` so hardware-backed signing and trust-tier policy are mandatory where production signing is allowed.
- Replace or quarantine the simulated KMS/Schnorr behavior in `src/enclave/cloud.rs` for production paths.
- Make `DeviceIntegrityReport::verify` in `src/enclave/attestation.rs` enforce the complete cryptographic evidence policy.
- Make `verify_hardware_integrity_with_policy` in `src/protocol/rails/mod.rs` fail closed for missing or insufficient evidence.

## Evidence
- [Audit PR #193](https://github.com/Conxian/conxius-enclave-sdk/pull/193) records simulated/software signing and incomplete attestation enforcement as P0 findings.
- Relevant implementation paths: `src/enclave/mod.rs`, `src/enclave/android_strongbox.rs`, `src/enclave/cloud.rs`, `src/enclave/attestation.rs`, and `src/protocol/rails/mod.rs`.
- Historical issue #180 documents a FROST treasury ceremony but does not provide this runtime enforcement or close the residual production boundary.

## Acceptance criteria
- Production configuration cannot instantiate a simulated or software signer for value-bearing signing or settlement operations.
- Full cryptographic attestation is enforced, including certificate-chain validation, freshness, nonce/replay protection, trusted-root validation, and purpose binding.
- Any attestation bypass is unavailable in production and remains limited to explicitly test-only harnesses.
- Negative tests reject missing, stale, replayed, wrong-root, wrong-purpose, and software-backed evidence without panic or permissive fallback.
- The implementation and support matrix state which hardware-backed capabilities are actually evidenced; unsupported capabilities remain disabled.

## Dependencies
- This is the first implementation gate and is a prerequisite for the canonical verification, protocol placeholder, WASM boundary, and final acceptance work in this backlog.
- Historical issue #180 remains related context only; it must not be treated as evidence that this enforcement is complete.
