# Architecture: align SDK policy types with Core control-model contracts

**Issue #194** | OPEN | Created: 2026-07-20 | Updated: 2026-07-20

## Summary

The completed SDK/Core alignment audit found an ownership and dependency gap: Core should own canonical protocol/control-model types, but the SDK currently carries duplicate or incompatible representations for `TrustTier`, chain and rail policy, replay, and BIP-110 concepts. The SDK has no dependency on Core, while Core currently optionally depends on `conxius-enclave-sdk` in its Cargo manifest; adding a direct reverse dependency from the SDK to Core would risk a Cargo dependency cycle.

This issue tracks the architecture decision and adapter work. It does not duplicate the SDK BIP-322 preimage-vs-digest work, which remains owned by [SDK #179](https://github.com/Conxian/conxius-enclave-sdk/issues/179).

## Evidence

- Core’s canonical trust contract in [`src/control_model/trust.rs`](https://github.com/Conxian/lib-conxian-core/blob/main/src/control_model/trust.rs) defines `TrustTier::{Strict, Managed, Expedient, ObserverOnly}`.
- The SDK’s rail policy in [`src/protocol/rails/mod.rs`](https://github.com/Conxian/conxius-enclave-sdk/blob/main/src/protocol/rails/mod.rs) defines `TrustTier::{T1, T2, T3, T4}`, which is a different model with SDK-specific runtime semantics.
- [Core PR #184](https://github.com/Conxian/lib-conxian-core/pull/184) canonicalizes serializable `Bip110Limits` and `Bip110TransactionShape`, including the exact 256-byte pushdata, 83-byte `OP_RETURN`, 34-byte ScriptPubKey, and 256-byte witness limits.
- The SDK duplicates BIP-110 limits and validation in [`src/protocol/bip110.rs`](https://github.com/Conxian/conxius-enclave-sdk/blob/main/src/protocol/bip110.rs). The BIP-322 preimage-vs-digest bug/work remains tracked separately in [SDK #179](https://github.com/Conxian/conxius-enclave-sdk/issues/179).
- Core currently declares an optional `conxius-enclave-sdk` dependency in [`Cargo.toml`](https://github.com/Conxian/lib-conxian-core/blob/main/Cargo.toml#L44-L49), so the dependency direction must be resolved before introducing any reverse edge.
- Parent research and alignment context: [Core #173](https://github.com/Conxian/lib-conxian-core/issues/173).

## Scope

- Decide dependency direction and the source of truth for shared protocol/control-model types.
- Define an explicit adapter/mapping for SDK-specific runtime policy, including rail and network policy; fail closed for unmapped values.
- Add serde, conversion, and mapping tests, including invalid and unmapped cases.
- Consume the canonical BIP-110 limits/transaction-shape contract without introducing a Cargo dependency cycle.
- Clarify ownership and boundaries for chain, rail, trust, replay, and BIP-110 semantics.

## Non-goals

- Move SDK runtime rail or network policy into Core.
- Implicitly change trust semantics.
- Remove or rename public SDK types without a migration path.
- Re-open or duplicate the BIP-322 preimage-vs-digest work owned by [SDK #179](https://github.com/Conxian/conxius-enclave-sdk/issues/179).

## Acceptance criteria

- [ ] Ownership and dependency direction are documented, including the cycle-safe integration boundary.
- [ ] SDK/Core mappings are explicit, tested, and fail closed for unmapped values.
- [ ] Serde round-trip and negative mapping tests cover the shared control-model values.
- [ ] SDK-facing BIP-110 validation consumes one canonical limits/transaction-shape source; divergent duplicate limits are removed or reduced to a tested adapter.
- [ ] Cargo metadata/build graph contains no dependency cycle.
- [ ] The migration path preserves existing public SDK behavior unless an explicit, reviewed compatibility decision says otherwise.
