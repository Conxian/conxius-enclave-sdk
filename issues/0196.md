# [P0] Implement canonical Bitcoin and Ethereum verification and derivation

**Issue #196** | OPEN | Created: 2026-07-20 | Updated: 2026-07-20

The audit identifies acceptance-only BIP-322 and non-canonical Bitcoin/Ethereum hashing and derivation as P0 blockers. Verification and derivation errors can produce incorrect authorization decisions, so the affected network capabilities remain conditional until canonical behavior is independently checked.

Refs #191

## Scope
- Implement actual BIP-322 verification in `src/protocol/bip322.rs` for every supported address and script form, rather than acceptance-only or structural validation.
- Correct BIP-340/341 tagged-hash, taproot tweak, and related derivation behavior in `src/protocol/bitcoin.rs`.
- Correct Keccak, EIP-specific behavior, and input validation in `src/protocol/ethereum.rs`.
- Ensure malformed, ambiguous, and out-of-range inputs return typed errors without panic, silent normalization, or default values.

## Evidence
- [Audit PR #193](https://github.com/Conxian/conxius-enclave-sdk/pull/193) records acceptance-only BIP-322 verification and non-canonical hashing as P0 findings.
- Relevant implementation paths: `src/protocol/bip322.rs`, `src/protocol/bitcoin.rs`, and `src/protocol/ethereum.rs`.
- Open issue #179 and merged [PR #189](https://github.com/Conxian/conxius-enclave-sdk/pull/189) address BIP-110 limits and BIP-322 chunking/segmentation, but do not close this broader canonical verification and derivation scope.

## Acceptance criteria
- Official Bitcoin and Ethereum vectors pass, and negative vectors cover invalid signatures, domains, encodings, addresses, scripts, hashes, and chain identifiers.
- Results are cross-checked against an independent reference implementation or independently generated vectors for each supported behavior.
- Taproot/BIP-340/341 tagged hashes and tweaks match their canonical specifications, including edge cases.
- Keccak and EIP input handling match canonical behavior for supported networks and reject malformed input deterministically.
- No malformed input path panics, silently truncates, or falls back to a default verification or derivation result.

## Dependencies
- Depends on the hardware-backed signing and attestation gate created in #195 for production authorization paths.
- Coordinates with the fail-closed asset and address work; the final acceptance gate must attach evidence for each supported chain.
