# fix(release): make toolchain and release evidence reproducible

**PR #213** | OPEN | Author: charliecreates[bot] | Created: 2026-07-20 | Merged: never

Branch: `fix/199-reproducible-release-supply-chain` → `main`

## Scope

This draft implements the reproducible-release and supply-chain hardening requested by #199 under umbrella #191. It is intentionally limited to repository controls and evidence plumbing; it does not perform a release or publish any package.

## What changed

- Track `Cargo.lock`, remove the ignore rule, pin the Rust 1.97.1 CI/release toolchain, and declare the verified Rust 1.94.1 MSRV.
- Apply `--locked` consistently where Cargo accepts it, keep `cargo-sbom` behind an explicit locked metadata gate, and use the valid `cargo audit --file Cargo.lock` form.
- Consolidate release behavior in `.github/workflows/release-strict.yml`: one tag-triggered crates.io publisher with the `release` environment, the same job as a manual recovery path, and one GitHub Release creator. Remove the conflicting legacy release/provenance/publish workflows.
- Generate and retain a crate checksum, SPDX 2.3 SBOM, release manifest, and GitHub build provenance; verify package metadata, lockfile hash, exact crate hash, manifest consistency, and provenance signer workflow before publication.
- Replace the placeholder secret-scan check with a full-history, checksum-verified Gitleaks scan on an explicit x86_64 runner.
- Update release/environment documentation and capability evidence without making a production-readiness claim.
- Apply the Rust 1.97 Clippy-required `is_multiple_of` changes in the BIP-110/BIP-322 validation paths.

## Non-claims

- This PR does not create a tag, publish to crates.io/NPM, create a GitHub Release, or change package version `2.0.12`.
- Repository metadata still declares `2.0.12` while the latest visible release/tag evidence remains `v2.0.11`; this PR does not reconcile that release history.
