# fix: harden telemetry privacy and delivery semantics

**PR #210** | OPEN | Author: charliecreates[bot] | Created: 2026-07-20 | Merged: never

Branch: `fix/201-telemetry-privacy` → `main`

## Summary

Remediates the telemetry privacy and delivery gaps identified by #201 and the production-enablement umbrella #191.

- Replaces credential- and signature-derived payload fields with a coarse, versioned event payload.
- Keeps credentials in a sensitive transport header only; no telemetry payload or diagnostics contain credentials, private keys, raw attestations, or request-derived identifiers.
- Keeps telemetry disabled by default and preserves the legacy `track_signature` API as a deprecated compatibility shim that discards its identifier argument.
- Validates production endpoints as HTTPS and fails closed on invalid compatibility configuration.
- Adds bounded timeout, retry, exponential backoff, and safe delivery status/failure observability without coupling telemetry to signing or settlement.
- Adds a `cfg(test)` request-capturing transport and public-safe operations guidance.

## Privacy model

The emitted body contains only `schema_version` and a coarse event name. API credentials are placed in a sensitive `X-Api-Key` header when configured and are never serialized into the body or logged. The SDK does not accept raw keys, signatures, attestations, addresses, assets, or request metadata as telemetry fields.

## Delivery semantics

- Explicit opt-in through `RailProxy::with_telemetry`; otherwise no telemetry is scheduled.
- Native scheduling outside an active Tokio runtime is rejected and recorded without panic.
- Five-second default request timeout, bounded to 30 seconds.
- Two retries by default, bounded to three, with bounded exponential backoff.
- Retries are limited to transport failures and transient HTTP statuses.
- `delivery_status`, `last_failure`, and `failure_count` expose safe local outcomes.
- Delivery is best effort; no durable queue or shutdown flush is implied, and telemetry never gates security-critical operations.

## Compatibility and operational risks

- Existing `TelemetryClient::new` and `track_signature` entry points remain available; invalid configuration fails closed, and the deprecated signature argument is intentionally ignored.
- The telemetry backend must accept the minimized event schema and header-based authentication; no backend contract or retention policy is asserted here.
- In-flight events can be lost during runtime/process shutdown. Telemetry remains non-authoritative and must not be used as a settlement or signing gate.
- This PR does not claim independent review, deployment evidence, release acceptance, or production support.
