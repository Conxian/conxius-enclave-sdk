# fix: canonicalize Bitcoin and Ethereum validation

**PR #208** | CLOSED | Author: charliecreates[bot] | Created: 2026-07-20 | Merged: 2026-07-20

Branch: `fix/196-canonical-bitcoin-ethereum` → `main`

## Summary

This draft implements the Bitcoin and Ethereum portions of #196 without modifying `src/protocol/bip322.rs`.

### Covered acceptance criteria

- **Canonical Bitcoin Taproot/BIP-340/341 derivation:** replaced the placeholder TapTweak midstate with `bitcoin::taproot::TapTweakHash::from_key_and_merkle_root`, validated x-only/compressed/uncompressed internal keys, rejected malformed keys without slicing or fallback, enforced the canonical BIP-86 path shape and Bitcoin coin types, and failed closed for out-of-range tweak scalars.
- **Bitcoin reference and negative vectors:** added the official BIP-86 internal-key/tweak/output-address vector plus malformed key, non-BIP-86 path, and unsupported-network negative coverage.
- **Canonical Ethereum address derivation:** replaced SHA-256 with Alloy Keccak-256 over the 64-byte uncompressed SEC1 public-key body and returned the canonical EIP-55 checksummed address.
- **Canonical Ethereum EIP behavior:** uses Alloy's EIP-191 personal-sign hash, validates returned ECDSA signatures against the returned enclave public key, and provides typed signature verification for compact and recovery-id forms.
- **Ethereum input validation:** rejects malformed public keys, addresses, contracts, signature encodings, recovery IDs, and mismatched signatures deterministically; ERC-20 preparation no longer uses default values or a panic-prone permissive slice.
- **Reference and negative vectors:** covers the canonical private-key-1 Ethereum address, the EIP-191 `Hello World` hash, malformed ERC-20 inputs, malformed signatures, and message/signature mismatches.

## BIP-322 / PR #206 dependency

BIP-322 remains explicitly out of scope for this branch. PR [#206](https://github.com/Conxian/conxius-enclave-sdk/pull/206) owns the related `src/protocol/bip322.rs` path and is already merged; this PR deliberately does not edit that file. Full canonical BIP-322 verification for every supported address and script form remains a separate acceptance item for #196.

## Remaining limitations

- This draft does not close the full issue: actual BIP-322 verification remains outside this branch and is not claimed here.
- Production authorization still depends on the hardware-backed signing and attestation gate in #195 and the repository's evidence chain; software/cloud signing paths remain conditional.
- The transaction-hash API accepts an already-hashed 32-byte payload and has no chain-ID parameter. EIP-155 transaction/domain separation and serialized transaction validation remain outside this focused change.
- This PR provides canonical derivation, validation, and fail-closed boundaries; it does not make a repository-wide production-support or release claim.
