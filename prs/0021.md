# Remediate Zero Secret Egress violation in Enclave layer

**PR #21** | CLOSED | Author: botshelomokoka | Created: 2026-04-04 | Merged: 2026-04-05

Branch: `fix/zero-secret-egress-violation-8409124210241316219` → `master`

### Selected task
Remediate the "Zero Secret Egress" violation in the Enclave layer.

### Risk addressed
Public/private boundary risk (Priority 1). The `generate_key` function in the `CoreEnclaveManager` was returning the raw 32-byte secret seed to the caller, violating the core security principle that private keys must never leave the hardware enclave.

### Evidence found
In `src/enclave/android_strongbox.rs`, the `generate_key` implementation was:
```rust
fn generate_key(&self, _key_id: &str) -> ConclaveResult<String> {
    let mut seed = [0u8; 32];
    rand::rng().fill_bytes(&mut seed);
    let key_hex = hex::encode(seed);
    seed.zeroize();
    Ok(key_hex)
}
```
Although `seed.zeroize()` was called, the raw seed was already encoded into `key_hex` and returned to the caller, resulting in a complete exposure of the private material.

### Changes made
- Modified `src/enclave/android_strongbox.rs` to derive a `secp256k1::PublicKey` from the generated seed.
- Updated the return value of `generate_key` to be the hex-encoded public key.
- Ensured the seed is zeroized immediately after public key derivation.
- Added necessary `secp256k1::PublicKey` import to the file.
- Added documentation comments explaining the security fix and its relationship to the StrongBox hardware model.

### Validation
- Successfully ran `cargo test` to ensure that the change does not break existing protocol or enclave tests.
- Verified that all 9 tests passed, including `protocol::business::tests::test_attribution_verification`.

### Documentation updated
- Added inline code comments in `src/enclave/android_strongbox.rs` detailing the remediation and the "Zero Secret Egress" principle.

### Follow-ups
Users of the SDK who utilized the `generate_key` function in version 0.1.0 (or prior to this fix) must consider any generated keys as compromised and perform a full key rotation.

### Approval note
This change is a critical security fix. Secret rotation is not required for the repository itself, but downstream users should be advised to rotate keys generated via this SDK method. Visibility review is recommended for the `generate_key` API to ensure it aligns with the intended production hardware handles.
