# fix: fail closed on unverifiable attestation and rail signing

**PR #214** | CLOSED | Author: charliecreates[bot] | Created: 2026-07-20 | Merged: 2026-07-20

Branch: `charlie/195-fail-closed-containment` → `main`

## Summary

This PR publishes the verified, rebased containment slice for the fail-closed work without overwriting concurrent PR #205.

- Adds a versioned, signed attestation envelope: report type, version, and the security-relevant fields are covered by the signature.
- Enforces exact typed policy tokens for the requested operation purpose, algorithm, provider, and trust requirements.
- Keeps the production provider verifier intentionally unavailable; unverifiable provider evidence cannot become an acceptance path.
- Removes the runtime bypass instead of treating software or missing-provider verification as valid production evidence.
- Makes replay saturation fail closed when no capacity is available for a new attestation.
- Corrects raw BIP340 signing to use the required raw signature encoding.
- Uses canonical complete-intent hashing that binds the rail and dispatch/security context.
- Seals rail execution behind the attested complete intent and rejects production raw broadcast before any network dispatch.
- Adds production-mode tests covering provider-unavailable verification, complete-intent binding, legacy hash rejection, and raw-broadcast rejection.

## Scope and limitations

This is containment, not production enablement. Real Android StrongBox, Nitro, Intel DCAP, and AMD SEV verification, together with typed operation key/algorithm/provider binding, remain required before production acceptance. These changes do not establish repository-wide production readiness or support.

## Compatibility notes

- Pre-versioned attestation reports are rejected.
- The insecure direct rail API is sealed.
- Request-only hashes are rejected; callers must use the canonical complete-intent hash.
- Raw broadcast is deprecated/unsupported in production and is rejected at the production boundary.
