# fix: harden CloudEnclave and ISO 20022 validation paths

**PR #36** | CLOSED | Author: charliecreates[bot] | Created: 2026-04-16 | Merged: 2026-04-16

Branch: `charlie/con-409-followups` → `master`

### Summary
Strengthen CloudEnclave safety guarantees and tighten ISO 20022 validation to reduce the risk of accepting malformed or unverified data, addressing feedback from PR #31.

### Changes
- **CloudEnclave**
  - Enforce stricter input validation on enclave-bound payloads (length, encoding, and allowed character sets).
  - Add explicit checks for missing or inconsistent enclave configuration before attempting operations.
  - Ensure all error paths surface deterministic and non-leaky errors (no internal state or secrets in messages).
  - Guard against silently accepting partial or truncated data by validating payload sizes against expected schema.

- **ISO 20022 validation**
  - Tighten schema validation to reject:
    - Fields with invalid character sets or disallowed whitespace.
    - Amount fields that exceed configured precision/scale or violate domain constraints.
    - Messages with missing mandatory elements and unexpected/unknown elements in strict mode.
  - Normalize and centralize ISO 20022 validation logic to reduce divergence between code paths.
  - Add explicit checks around optional vs mandatory fields to eliminate ambiguity in parsing.

- **Error handling & logging**
  - Standardize error types for CloudEnclave and ISO 20022 validation failures to make handling and observability consistent.
  - Reduce log verbosity on expected validation failures while ensuring enough detail for debugging (non-secret only).

- **Tests**
  - Extend unit tests to cover new negative cases for CloudEnclave payloads (invalid sizes, malformed content, unsupported configs).
  - Add ISO 20022 tests for malformed, truncated, and over-precision amounts, plus invalid character sets and missing mandatory fields.
  - Ensure tests reflect the tightened validation semantics requested in PR #31 review.

### Motivation
These changes respond to requested hardening in PR #31 (review by @botshelomokoka) to:
- Prevent accepting malformed enclave payloads or ISO 20022 messages.
- Standardize error behavior across validation paths.
- Improve safety and predictability of enclave-related operations before production readiness.

### Notes
- No breaking API changes are introduced; behavior differences are confined to stricter validation and error responses.
- Downstream consumers that previously relied on permissive validation may now see failures where invalid data was previously accepted, which is the desired outcome for security and correctness.
